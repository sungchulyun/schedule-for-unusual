# ScheduleApp 백엔드 서버 요구사항 명세서

## 1. 문서 목적

이 문서는 현재 Android `ScheduleApp`의 월간 캘린더 기능을 서버 기반 구조로 확장하기 위한 백엔드 요구사항을 정의한다.

본 문서에서의 우선순위 원칙은 아래와 같다.

- 초기 MVP는 `인증/인가`와 `캘린더 데이터 저장/조회 기능`을 함께 구현한다.
- 즉, 카카오 OAuth2 로그인, JWT 발급/재발급, 일정/근무 스케줄 저장, 월간 조회, 날짜 상세 조회가 1차 범위다.
- 초대와 커플 연결 고도화는 인증 이후 단계로 확장한다.

## 1.1 확정 기술 스택

백엔드 구현 스택은 아래와 같이 확정한다.

- API 서버: `Spring Boot`
- 인증/인가: `Spring Security`, `JWT`
- 소셜 로그인: `Kakao OAuth2`
- 푸시 알림 플랫폼: `Firebase Cloud Messaging (FCM)`
- 주 데이터베이스: `MySQL`

역할 분담 원칙은 아래와 같다.

- `Spring Boot`는 REST API, 비즈니스 로직, 검증, 트랜잭션, 데이터 접근을 담당한다.
- `MySQL`은 일정, 근무 스케줄, 그룹, 사용자 등 정식 서비스 데이터를 저장하는 주 저장소로 사용한다.
- `FCM`은 핵심 데이터 저장소가 아니라, 푸시 알림 전달 채널로만 사용한다.
- 분석, 원격 설정, Crashlytics 등 다른 Firebase 제품은 현재 범위에 포함하지 않는다.

현재 앱은 아래 기능을 로컬 상태로 처리하고 있다.

- 내 일정
- 우리 일정
- 상대 일정
- 날짜별 근무 스케줄
- 월간 달력 조회
- 날짜 상세 조회
- 일정 추가/수정/삭제
- 날짜별 또는 월별 근무 스케줄 등록

백엔드 서버 도입 목적은 다음과 같다.

- 앱 데이터를 영구 저장한다.
- 커플/공유 사용자 간 동일 데이터를 동기화한다.
- 여러 기기에서 동일한 캘린더 상태를 조회/수정할 수 있게 한다.
- 향후 FCM 푸시 알림, 초대, 실시간 반영, 웹/관리자 기능 확장 기반을 마련한다.

## 2. 범위

### 2.1 1차 서버 구축 범위

1차 서버 범위는 아래 기능까지 포함한다.

- 카카오 OAuth2 기반 회원가입/로그인
- JWT 액세스 토큰/리프레시 토큰 발급, 재발급, 로그아웃
- 일정 CRUD
- 근무 스케줄 CRUD
- 월간 캘린더 조회
- 특정 날짜 상세 조회
- 필터링에 필요한 데이터 제공
- 최소 실행 환경용 식별 체계
- FCM 등록 토큰 관리
- FCM 기반 일정 변경 및 오늘 일정 요약 알림 발송

초기 MVP 기준으로는 아래 순서를 따른다.

1. 카카오 OAuth2 + JWT 인증 API
2. 캘린더 데이터 저장/조회 API
3. 월별 스케줄 일괄 저장 API
4. 캘린더 통합 조회 API
5. 이후 커플 연결 기능

초기 MVP 기준 저장 구조는 다음을 원칙으로 한다.

- 캘린더 핵심 데이터는 `MySQL`에 저장한다.
- 서버 API는 `Spring Boot`로 제공한다.
- 인증/인가 처리는 `Spring Security + Kakao OAuth2 + JWT`를 사용한다.

### 2.2 1차 서버 범위 제외

아래 항목은 초기 MVP 출시 범위에서 제외하거나 후순위로 둔다.

- 커플 또는 공유 그룹 연결
- 초대 링크 또는 초대 코드
- 그룹 멤버 관리

- 실시간 소켓 동기화
- 반복 일정
- 파일 첨부
- 관리자 콘솔
- 다자간 그룹 캘린더
- 외부 캘린더 연동

## 3. 이해관계자

- 앱 사용자
- 앱 개발자
- 백엔드 개발자
- Android 클라이언트 개발자
- QA 담당자

## 4. 핵심 용어 정의

- 사용자: 앱에 로그인하는 개인 계정
- 커플 그룹: 두 사용자가 공유하는 캘린더 단위
- 사용자 한 명은 동시에 하나의 커플 그룹에만 속할 수 있다.
- 일정: 제목, 기간, 주체, 메모를 가지는 캘린더 항목
- 일정 주체:
  - `ME`
  - `US`
  - `PARTNER`
- 근무 스케줄: 특정 날짜의 근무 유형
- 월간 조회: 특정 연월의 일정 및 스케줄 목록 조회
- 날짜 상세 조회: 특정 날짜에 걸치는 일정 및 스케줄 조회

## 5. 시스템 목표

백엔드 시스템은 아래를 만족해야 한다.

1. 일정과 근무 스케줄은 서버에 저장되어 앱 재설치 후에도 유지되어야 한다.
2. 멀티데이 일정은 시작일과 종료일 기준으로 저장되어야 한다.
3. 앱은 월간 화면과 날짜 상세 화면에 필요한 데이터를 빠르게 가져와야 한다.
4. 초기 MVP에서는 단일 사용자 또는 임시 식별 기준으로도 서버 저장/조회가 가능해야 한다.
5. 인증/인가가 추가되더라도 API 구조를 크게 깨지 않고 확장 가능해야 한다.

## 6. 기능 요구사항

### 6.1 계정 및 인증

이 절의 기능은 초기 MVP 필수 구현 범위다.

#### FR-001 사용자 가입

- 사용자는 카카오 OAuth2 인증을 통해 가입할 수 있어야 한다.
- 최초 카카오 로그인 시 서버는 자동 회원가입을 처리할 수 있어야 한다.
- 서버는 카카오 사용자 식별값을 내부 사용자 계정과 매핑 저장해야 한다.

#### FR-002 로그인

- 사용자는 카카오 OAuth2 인증을 통해 로그인할 수 있어야 한다.
- 로그인 성공 시 액세스 토큰과 리프레시 토큰을 발급해야 한다.
- 인증 처리는 `Spring Security` 기반으로 일관되게 수행해야 한다.

#### FR-003 토큰 갱신

- 클라이언트는 만료된 액세스 토큰을 리프레시 토큰으로 갱신할 수 있어야 한다.
- 리프레시 토큰은 서버에서 무효화 가능해야 한다.

#### FR-004 로그아웃

- 사용자는 현재 세션 또는 전체 세션 로그아웃을 할 수 있어야 한다.
- 로그아웃 시 대상 리프레시 토큰은 재사용 불가 상태가 되어야 한다.

#### FR-005 내 정보 조회

- 로그인된 사용자는 자신의 프로필 정보를 조회할 수 있어야 한다.
- 응답에는 최소 `userId`, `nickname`, `oauthProvider`, `defaultShiftOwnerType`이 포함되어야 한다.

#### FR-006 사용자 표시 설정 수정

- 로그인된 사용자는 캘린더 표시 설정을 수정할 수 있어야 한다.
- 초기 설정 항목은 기본 근무 스케줄 표시 대상(`defaultShiftOwnerType`)을 포함한다.
- `defaultShiftOwnerType`은 `ME` 또는 `PARTNER`만 허용한다.

### 6.2 커플 또는 공유 관계 관리

이 절의 기능은 카카오 로그인과 일정/근무 스케줄 기본 기능 이후 단계에서 구현한다.
다만 현재 프로젝트 방향 기준으로는 `카카오톡 공유 기반 초대 링크 -> 상대방 가입/로그인 -> 초대 수락` 흐름을 표준 방식으로 삼는다.

#### FR-010 커플 그룹 생성

- 최초 가입 사용자는 개인 상태로 시작하거나 커플 그룹을 생성할 수 있어야 한다.
- 가입 직후 사용자는 파트너가 없는 단독 그룹 상태로 시작할 수 있어야 한다.

#### FR-011 초대 생성

- 사용자는 상대방을 초대할 수 있는 초대 링크를 생성할 수 있어야 한다.
- 초대 링크에는 초대 토큰 또는 초대 식별자가 포함되어야 한다.
- 파트너 연결은 초기 구현 기준으로 `카카오톡 공유` 기반 흐름을 우선 지원해야 한다.
- 신규 유입 사용자 초대 목적에서는 카카오 친구 메시지 API 직접 발송이 아니라 사용자가 직접 대상을 선택하는 카카오톡 공유 방식을 기본으로 사용해야 한다.
- 서버는 초대 링크 생성 시 만료 시간, 발신자, 대상 그룹, 초대 상태를 저장할 수 있어야 한다.

#### FR-012 초대 메시지 공유

- 사용자는 생성된 초대 링크를 카카오톡 공유로 전달할 수 있어야 한다.
- 앱은 카카오톡 공유용 제목, 설명, 버튼 링크를 구성할 수 있어야 한다.
- Android 앱의 기본 진입점은 메인 캘린더 상단 배너형 `파트너 초대` CTA여야 한다.
- 상단 배너는 파트너 미연결 상태에서만 노출하고, 일정 입력 액션과 구분되어야 한다.
- 공유 메시지의 링크는 앱 딥링크 또는 웹 링크를 통해 초대 상세 화면으로 연결될 수 있어야 한다.
- 공유 링크는 카카오 디벨로퍼스의 Product Link 설정과 앱 스킴 설정 범위 안에서 동작해야 한다.

#### FR-013 초대 링크 진입

- 상대방 사용자는 카카오톡 메시지의 초대 링크를 눌러 앱 또는 웹 진입 지점으로 이동할 수 있어야 한다.
- 비로그인 상태에서는 초대 토큰을 유지한 채 회원가입 또는 로그인으로 이어질 수 있어야 한다.
- 로그인 완료 후에는 저장된 초대 토큰을 기준으로 초대 상세 또는 수락 화면으로 복귀할 수 있어야 한다.

#### FR-014 초대 수락

- 상대방 사용자는 초대를 수락해 같은 커플 그룹에 참여할 수 있어야 한다.
- 초대 수락은 로그인 사용자 기준으로만 가능해야 한다.
- 만료되었거나 이미 사용되었거나 취소된 초대는 수락할 수 없어야 한다.
- 이미 다른 그룹에 속한 사용자는 초대를 수락할 수 없어야 한다.
- 이미 파트너가 연결된 그룹에 대한 초대는 수락할 수 없어야 한다.
- 초대 수락 성공 응답은 최신 `groupId`가 포함된 access/refresh token을 반환해야 한다.

#### FR-015 그룹 멤버 조회

- 사용자는 현재 자신이 속한 커플 그룹의 멤버 정보를 조회할 수 있어야 한다.
- 사용자는 동시에 하나의 커플 그룹에만 참여할 수 있어야 한다.

#### FR-016 파트너 연결 완료

- 초대 수락이 성공하면 초대한 사용자와 수락한 사용자는 같은 커플 그룹에 속해야 한다.
- 한 커플 그룹에는 본인을 포함해 최대 2명까지만 등록할 수 있어야 한다.
- 연결 완료 후 두 사용자 모두 파트너 연결 상태를 확인할 수 있어야 한다.

#### FR-017 파트너 연결 후 공동 편집 권한

- 파트너로 등록된 사용자는 같은 커플 그룹의 일정과 근무 스케줄을 함께 조회할 수 있어야 한다.
- 파트너로 등록된 사용자는 `ME`, `US`, `PARTNER` 주체 구분과 무관하게 그룹 내 모든 일정을 생성/수정/삭제할 수 있어야 한다.
- 파트너로 등록된 사용자는 그룹 내 근무 스케줄을 조회할 수 있어야 한다.
- 근무 스케줄 생성/수정/삭제는 기본적으로 현재 로그인 사용자의 스케줄에만 적용한다.
- 서버는 생성자와 수정자를 감사 로그로 구분할 수 있어야 한다.

### 6.3 일정 관리

이 절은 초기 MVP의 최우선 구현 범위다.

#### FR-020 일정 생성

- 사용자는 일정을 생성할 수 있어야 한다.
- 일정 입력 항목:
  - 제목
  - 시작일
  - 종료일
  - 일정 주체
  - 메모
- 서버는 일정 저장 시 작성 사용자 식별값을 `User.id` 기준으로 기록할 수 있어야 한다.

#### FR-021 일정 수정

- 사용자는 자신이 속한 커플 그룹의 일정을 수정할 수 있어야 한다.
- 파트너로 연결된 두 사용자 모두 그룹 내 `나`, `우리`, `상대` 일정 전체를 수정할 수 있어야 한다.

#### FR-022 일정 삭제

- 사용자는 자신이 속한 커플 그룹의 일정을 삭제할 수 있어야 한다.
- 파트너로 연결된 두 사용자 모두 그룹 내 `나`, `우리`, `상대` 일정 전체를 삭제할 수 있어야 한다.

#### FR-023 일정 조회

- 서버는 월간 조회 및 날짜 상세 조회에 필요한 일정 목록을 제공해야 한다.
- 서버는 현재 인증 사용자 기준으로 `ME`, `US`, `PARTNER` 표시값을 계산해 응답할 수 있어야 한다.

#### FR-024 멀티데이 일정 처리

- 시작일과 종료일이 다른 일정은 멀티데이 일정으로 저장되어야 한다.
- 날짜 상세 조회 시 해당 날짜에 걸치는 일정으로 포함되어야 한다.

### 6.4 근무 스케줄 관리

이 절은 초기 MVP의 최우선 구현 범위다.

#### FR-030 날짜별 스케줄 저장

- 사용자는 특정 날짜의 근무 스케줄을 저장할 수 있어야 한다.
- 커플 두 명이 모두 스케줄 근무를 수행하는 경우 각 사용자의 근무 스케줄은 별도로 저장되어야 한다.
- 같은 그룹, 같은 날짜라도 사용자별로 각각 하나의 근무 스케줄을 가질 수 있어야 한다.

#### FR-031 날짜별 스케줄 수정

- 사용자는 특정 날짜의 기존 스케줄을 변경할 수 있어야 한다.
- 사용자는 자신의 근무 스케줄을 수정할 수 있어야 한다.

#### FR-032 날짜별 스케줄 삭제

- 사용자는 특정 날짜의 스케줄을 삭제할 수 있어야 한다.
- 사용자는 자신의 근무 스케줄을 삭제할 수 있어야 한다.

#### FR-033 월별 일괄 저장

- 클라이언트는 한 달 단위의 스케줄을 일괄 저장할 수 있어야 한다.
- 저장 방식은 `현재 사용자의 해당 월 전체 덮어쓰기`를 기본으로 한다.

#### FR-034 근무 스케줄 타입

- 서버는 아래 스케줄 타입을 지원해야 한다.
  - `DAY`
  - `NIGHT`
  - `MID`
  - `EVENING`
  - `OFF`
  - `VACATION`

#### FR-035 기본 근무 스케줄 표시 대상 설정

- 사용자는 캘린더에서 기본으로 표시할 근무 스케줄 소유자를 설정할 수 있어야 한다.
- 설정값은 최소 `ME`, `PARTNER`를 지원해야 한다.
- 커플 중 한 명이 상근 등으로 근무 스케줄을 별도로 기록할 필요가 없는 경우, 해당 사용자는 기본 표시 대상을 `PARTNER`로 설정할 수 있어야 한다.
- 기본 표시 대상이 `PARTNER`이면 월간 캘린더 조회와 날짜 상세 조회에서 별도 파라미터가 없어도 상대방 근무 스케줄이 기본으로 반환되어야 한다.
- 사용자가 조회 요청에서 명시적으로 `shiftOwnerType`을 전달한 경우에는 요청 파라미터가 저장된 기본 설정보다 우선해야 한다.

### 6.5 캘린더 조회

이 절은 초기 MVP의 최우선 구현 범위다.

#### FR-040 월간 캘린더 조회

- 클라이언트는 특정 연월 기준으로 월간 캘린더 데이터를 조회할 수 있어야 한다.
- 응답에는 최소 아래 정보가 포함되어야 한다.
  - 해당 월에 걸치는 일정 목록
  - 선택된 사용자(ME 또는 PARTNER)의 해당 월 날짜별 근무 스케줄 목록
  - 멤버 정보 또는 주체 표시용 기본 메타데이터

#### FR-041 날짜 상세 조회

- 클라이언트는 특정 날짜의 일정 및 스케줄 상세를 조회할 수 있어야 한다.
- 해당 날짜에 걸치는 멀티데이 일정도 포함해야 한다.

#### FR-042 필터 지원

- 서버는 최소 필터링 가능한 구조를 제공해야 한다.
  - 일정 주체별 필터
  - 근무 스케줄 포함 여부
  - 달력 화면에 표시할 근무 스케줄 소유자 선택(ME 또는 PARTNER)
- 서버는 사용자의 기본 근무 스케줄 표시 대상 설정을 제공하거나 조회 시 반영할 수 있어야 한다.
- 다만 MVP에서는 서버 필터보다 클라이언트 필터를 우선 허용하고, 서버는 전체 데이터 응답도 가능하다.

### 6.6 변경 이력 및 동기화 보조

#### FR-050 수정 시각 관리

- 일정과 근무 스케줄은 `createdAt`, `updatedAt`을 가져야 한다.

#### FR-051 삭제 처리 방식

- MVP부터 `deletedAt` 기반 소프트 삭제를 적용해야 한다.
- 기본 조회에서는 `deletedAt`이 없는 데이터만 반환해야 한다.
- 운영상 필요한 경우에만 별도 정리 작업으로 물리 삭제할 수 있다.

## 7. 비기능 요구사항

### 7.1 성능

- 월간 캘린더 조회 API는 일반적인 단일 사용자 기준 1초 이내 응답을 목표로 한다.
- 날짜 상세 조회 API는 일반적인 단일 사용자 기준 500ms 이내 응답을 목표로 한다.

### 7.2 가용성

- MVP에서는 상시 고가용성 구성이 필수는 아니다.
- 단일 리전 배포를 허용한다.

### 7.3 보안

- 모든 API는 HTTPS 사용을 권장한다.
- Spring Boot 서버는 HTTPS 환경에서 배포되어야 한다.
- 액세스 토큰과 리프레시 토큰은 안전하게 저장/전달되어야 한다.
- 사용자는 자신이 속한 그룹 데이터만 접근 가능해야 한다.
- 파트너로 등록된 사용자는 같은 그룹의 전체 일정을 공동 편집하고, 근무 스케줄은 서로 조회할 수 있어야 한다.
- 초기 MVP부터 인증 사용자 기준 그룹 접근 제어를 적용해야 한다.
- JWT 서명 오류, 만료, 위변조는 명확한 인증 에러로 처리해야 한다.

### 7.4 확장성

- 추후 아래 기능이 추가 가능하도록 설계해야 한다.
  - 반복 일정
  - FCM 기반 푸시 알림
  - 실시간 동기화
  - 웹 클라이언트
  - 관리자 기능

### 7.5 감사 및 운영

- 최소한 아래 로그는 서버에 남겨야 한다.
  - 로그인 성공/실패
  - 토큰 재발급 성공/실패
  - 로그아웃
  - 일정 생성/수정/삭제
  - 스케줄 생성/수정/삭제
  - 초대 생성/수락

## 9. 데이터 제약사항

### 9.1 일정

- 제목은 필수다.
- 시작일은 종료일보다 늦을 수 없다.
- 저장 기준 `subjectType`은 `PERSONAL`, `SHARED` 중 하나여야 한다.
- `subjectType = PERSONAL`이면 `ownerUserId`가 반드시 있어야 한다.
- `subjectType = SHARED`이면 `ownerUserId`는 nullable 허용이다.

### 9.2 스케줄

- 같은 그룹, 같은 날짜, 같은 사용자 조합의 근무 스케줄은 1개만 허용한다.
- 근무 스케줄 타입은 정의된 enum만 허용한다.

### 9.3 그룹

- MVP에서는 한 커플 그룹의 멤버 수는 최대 2명으로 제한한다.
- 사용자는 동시에 하나의 커플 그룹에만 속할 수 있다.
- 동일 그룹에는 본인 외 파트너 1명만 추가할 수 있다.
- 파트너가 연결된 이후에는 두 멤버 모두 그룹 데이터를 수정할 수 있다.

## 12. 권한 요구사항

이 절은 초기 MVP부터 적용된다.

### 12.1 사용자 본인 접근

- 사용자는 자신의 계정 정보만 조회/수정할 수 있어야 한다.

### 12.2 그룹 데이터 접근

- 사용자는 자신이 속한 그룹의 일정과 스케줄만 접근할 수 있어야 한다.
- 파트너로 연결된 두 사용자는 동일 그룹의 일정과 스케줄을 모두 조회할 수 있어야 한다.
- 파트너로 연결된 두 사용자는 동일 그룹의 `ME`, `US`, `PARTNER` 일정 전체를 수정/삭제할 수 있어야 한다.
- 근무 스케줄 수정/삭제는 기본적으로 현재 로그인 사용자의 스케줄에만 허용한다.

### 12.3 초대 처리

- 초대 코드는 만료 시간을 가져야 한다.
- 만료되었거나 이미 사용된 초대 코드는 재사용할 수 없어야 한다.
- 초대 토큰 수락이 성공하면 클라이언트가 즉시 세션 토큰을 교체할 수 있도록 새 access/refresh token을 반환해야 한다.

### 12.4 파트너 등록 제약

- 파트너 등록 요청은 같은 그룹에 빈 파트너 슬롯이 있을 때만 성공해야 한다.
- 이미 파트너가 연결된 그룹에는 추가 사용자 등록을 허용하지 않아야 한다.
- 이미 다른 그룹에 속한 사용자는 파트너로 등록할 수 없어야 한다.
- 사용자는 자신이 속하지 않은 그룹에 파트너를 등록할 수 없어야 한다.

## 13. 동기화 및 클라이언트 연동 요구사항

### 13.1 클라이언트 초기 동작

- 앱 실행 후 현재 월 기준 캘린더 데이터를 한 번에 불러올 수 있어야 한다.
- 로그인 상태가 없으면 카카오 로그인 흐름으로 진입해야 한다.

### 13.2 월 이동

- 사용자가 이전/다음 달로 이동할 때 해당 월 데이터를 요청할 수 있어야 한다.

### 13.3 날짜 상세

- 특정 날짜 선택 시 해당 날짜 상세 데이터를 별도 조회하거나, 월간 응답 데이터에서 재사용할 수 있어야 한다.

### 13.4 저장 후 반영

- 일정/스케줄 저장 성공 시 서버는 최신 데이터를 반환하거나, 클라이언트가 재조회 없이 반영 가능한 최소 정보를 반환해야 한다.

## 14. 권장 아키텍처

확정 스택 기준 권장 구조는 아래와 같다.

- API Layer
- Service Layer
- Repository Layer
- Database Layer

인증은 초기 MVP의 필수 계층이다.

구성 권장안:

- `Spring Boot`
  - REST API 제공
  - 입력값 검증
  - 도메인 서비스 로직 처리
  - JPA 또는 MyBatis 기반 DB 접근

  - OAuth2 로그인 진입점 관리
  - JWT 인증 필터 처리

- `Kakao OAuth2`
  - 사용자 인증 위임
  - 사용자 프로필 조회

- `JWT`
  - 액세스 토큰 발급
  - 리프레시 토큰 발급 및 재발급
  - API 요청 인증 컨텍스트 구성

  - 일정, 근무 스케줄, 그룹, 사용자 등 핵심 데이터 영속화

  - FCM 전송은 `app.fcm.enabled=true`와 Firebase 서비스 계정 경로가 설정된 환경에서만 활성화한다.
  - 운영 기본 서비스 계정 경로는 `classpath:firebase/schedule-for-unusual-fbe03-firebase-adminsdk-fbsvc-f4d15576a0.json`이며, 필요하면 `APP_FCM_CREDENTIALS_PATH`로 덮어쓴다.
  - 현재 Docker 배포는 서버의 `/opt/schedule-api/firebase/schedule-for-unusual-fbe03-firebase-adminsdk-fbsvc-f4d15576a0.json`를 컨테이너의 `/app/firebase/firebase-adminsdk.json`로 read-only mount하고, `APP_FCM_CREDENTIALS_PATH=/app/firebase/firebase-adminsdk.json`를 주입한다.
  - 현재 서버 발송 대상은 일정 생성/수정/삭제 알림과 매일 오늘 일정 요약 알림이다.

MySQL을 주 저장소로 선택하는 이유:

- 날짜 조건 조회가 많다.
- 사용자/그룹/초대/일정/스케줄 관계가 명확하다.
- 제약조건 관리가 쉽다.
- Spring Boot와의 일반적인 조합이 안정적이다.

## 15. 권장 구현 우선순위

### 단계 1

## 16. 완료 기준

아래가 모두 가능하면 초기 MVP 서버 구축 완료로 본다.

1. 월간 캘린더 화면에 필요한 일정/스케줄 데이터를 서버에서 조회할 수 있다.
2. 사용자는 카카오 계정으로 회원가입/로그인할 수 있다.
3. 액세스 토큰과 리프레시 토큰이 정상 발급/재발급된다.
4. 일정 추가/수정/삭제가 서버에 반영된다.
5. 근무 스케줄 추가/수정/삭제 및 월별 일괄 저장이 서버에 반영된다.
6. 날짜 상세 조회가 가능하다.
7. 인증 사용자 기준 그룹 데이터 접근 제어가 가능하다.

아래 항목은 초기 MVP 완료 조건이 아니라 후속 단계 완료 조건이다.

1. 두 사용자는 하나의 공유 그룹으로 연결될 수 있다.
2. 파트너로 연결된 두 사용자는 같은 그룹의 일정을 함께 수정하고, 근무 스케줄은 서로 조회할 수 있다.
3. 권한 없는 사용자는 다른 그룹 데이터에 접근할 수 없다.

## 17. 미결정 사항

아래 항목은 최종 구현 전에 확정이 필요하다.

1. 카카오 로그인 후 기본 그룹을 즉시 생성할지 여부
2. 커플 그룹 생성 시 기본 표시명 정책
3. 월간 조회에서 필터를 서버가 처리할지, 클라이언트가 처리할지
4. 기본 근무 스케줄 표시 대상 설정을 사용자 단위로 저장할지, 커플 그룹 단위로 저장할지
5. 배포 환경을 단일 서버로 시작할지, 관리형 클라우드로 시작할지
6. 리프레시 토큰 저장 전략을 DB 기반으로 고정할지 여부
7. FCM을 초기 단계에서 바로 붙일지, 후속 단계로 둘지
