# ScheduleApp 파트너 초대 흐름 문서

## 1. 문서 목적

이 문서는 `카카오톡 공유 기반 초대 링크 -> 상대방 가입/로그인 -> 초대 수락 -> 파트너 연결 완료` 흐름을 별도로 정리한다.

본 문서는 아래 목적을 가진다.

- 파트너 초대 기능의 표준 사용자 흐름을 정리한다.
- 카카오 로그인 기능 문서와 초대 기능 문서를 분리해 관리한다.
- Android 앱, 백엔드, 문서 작업 범위를 초대 기능 기준으로 명확히 한다.
- 카카오 친구 메시지 직접 발송이 아닌 `카카오톡 공유` 기반 구현 제약을 명확히 남긴다.

## 2. 기본 원칙

현재 프로젝트에서 파트너 초대는 아래 방식을 기본으로 사용한다.

- 서버의 카카오 친구 메시지 API 직접 발송이 아니라 `카카오톡 공유`를 사용한다.
- 초대 링크에는 초대 토큰 또는 초대 식별자가 포함된다.
- 상대방은 초대 링크로 진입한 뒤 가입 또는 로그인 후 초대를 수락한다.
- 초대 수락이 완료되면 두 사용자는 같은 그룹으로 연결된다.

이 방식을 선택하는 이유는 아래와 같다.

- 신규 유입 사용자에게도 초대 링크 전달이 가능하다.
- 사용자가 카카오톡에서 직접 대상을 선택하므로 정책 제약을 피하기 쉽다.
- Android 앱 UX와 카카오 SDK 로그인 구조에 더 잘 맞는다.

## 3. 사용자 흐름

### 3.1 초대 생성자 흐름

1. 로그인 사용자가 메인 캘린더 상단의 `파트너 연결이 아직 없어요` 배너에서 `파트너 초대` 버튼을 누른다.
2. 앱은 `POST /api/v1/groups/invites`를 호출해 초대 토큰과 공유 링크를 발급받는다.
3. 앱은 카카오톡 공유 메시지를 구성한다.
4. 사용자는 카카오톡에서 대상을 선택해 초대 메시지를 보낸다.

상단 배너 노출 원칙:

- 배너는 `파트너 미연결 상태`에서만 노출한다.
- 이미 파트너가 연결된 이후에는 초대 CTA를 기본 화면에서 숨긴다.
- 초대 기능은 일정 등록용 FAB나 날짜 셀 액션이 아니라 `그룹 연결 상태 액션`으로 취급한다.

### 3.2 초대 수신자 흐름

1. 상대방이 카카오톡 메시지의 초대 링크를 누른다.
2. 앱 설치 여부에 따라 앱 또는 웹 진입 지점으로 이동한다.
3. 비로그인 상태이면 초대 토큰을 유지한 채 가입 또는 로그인으로 이동한다.
4. 로그인 완료 후 초대 상세 화면으로 복귀한다.
5. 상대방이 `수락` 버튼을 누른다.
6. 앱은 `POST /api/v1/groups/invites/accept`를 호출한다.
7. 앱은 응답의 새 access/refresh token으로 세션을 교체한다.
8. 성공 시 두 사용자는 같은 그룹으로 연결된다.

## 4. 백엔드 책임

백엔드는 아래 책임을 가진다.

- 초대 생성 API 제공
- 초대 조회 API 제공
- 초대 수락 API 제공
- 초대 토큰 만료/재사용 방지
- 발신자, 대상 그룹, 상태 저장
- 초대 생성자의 자기 초대 수락 차단
- 이미 다른 그룹에 속한 사용자 차단
- 이미 파트너가 연결된 그룹 차단
- 초대 수락 성공 시 최신 `groupId`가 포함된 토큰 재발급

권장 API는 아래와 같다.

- `POST /api/v1/groups/invites`
- `GET /api/v1/groups/invites/{inviteToken}`
- `POST /api/v1/groups/invites/accept`

## 5. Android 앱 책임

Android 앱은 아래 책임을 가진다.

- 메인 캘린더 상단 배너에서 초대 CTA를 노출
- 초대 생성 API 호출
- 카카오톡 공유 템플릿 구성
- 공유 링크 전달
- 초대 링크 진입 처리
- 비로그인 상태에서 초대 토큰 임시 보관
- 로그인 완료 후 초대 상세 화면 복귀
- 초대 수락 API 호출
- 연결 완료 UI 표시

## 6. 데이터 및 상태 규칙

초대 기능은 최소한 아래 상태를 가져야 한다.

- `PENDING`
- `ACCEPTED`
- `CANCELLED`
- `EXPIRED`

추가 규칙:

- 초대 링크는 만료 시간을 가져야 한다.
- 같은 그룹에 아직 만료되지 않은 `PENDING` 초대가 있으면 새 초대 토큰을 발급하지 않고 기존 초대 정보를 재사용한다.
- 만료된 `PENDING` 초대가 있으면 `EXPIRED`로 정리한 뒤 새 초대를 생성한다.
- 초대 생성자는 본인이 만든 초대를 수락할 수 없어야 한다.
- 수락은 1회성으로 처리되어야 한다.
- 이미 사용된 초대는 재사용할 수 없어야 한다.
- 이미 다른 그룹에 속한 사용자는 수락할 수 없어야 한다.
- 이미 파트너가 연결된 그룹은 추가 수락을 허용하면 안 된다.

## 7. 카카오톡 공유 메시지 구성

초대 메시지는 아래 정보를 포함하는 것이 적절하다.

- 초대한 사용자 닉네임
- 서비스 설명 또는 연결 안내 문구
- 초대 수락 버튼
- 앱 실행 또는 웹 fallback 링크

예시 메시지 개념:

- 제목: `성철님이 파트너 연결을 요청했어요`
- 설명: `ScheduleApp에서 캘린더를 함께 쓰려면 초대를 수락해 주세요.`
- 버튼 링크: `https://app.example.com/invites/{inviteToken}`

## 8. UI 배치 원칙

초대 버튼 위치는 아래 기준으로 고정한다.

- 기본 진입점은 메인 캘린더 상단 배너형 UI다.
- 배너 문구는 파트너 미연결 상태를 설명하고, 버튼 라벨은 `파트너 초대`로 유지한다.
- 하단 플로팅 버튼, 날짜 셀, 일정 입력 버튼 주변에는 초대 CTA를 두지 않는다.
- 설정 화면이나 그룹 관리 화면은 보조 진입점으로 둘 수 있지만, 초기 MVP의 대표 진입점은 상단 배너다.
